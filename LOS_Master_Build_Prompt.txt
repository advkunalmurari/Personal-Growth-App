MISSION BRIEF
Build a Life Operating System


You are building a high-performance personal productivity application that functions as a Life Operating System (LOS). It converts long-term ambitions into daily measurable actions through goal hierarchies, automated task breakdown, habit tracking, and intelligent feedback loops.

Core Engineering Insight
People fail not because their goals are too big — but because their feedback loops are too weak. This system exists to manufacture constant, visible progress. Small daily steps. Relentless iteration. Compounding outcomes.

SECTION 3
Application Modules


The application has 6 core modules. Each must be a distinct route or view. All modules share state via global store (Zustand). Build in this order.

Module 1 — Life Dashboard  /dashboard
The command center. Renders immediately on login. Zero loading states — all data must be pre-fetched via SSR or optimistic cache.

Required Components
	•	TodayPanel: Ordered list of today's tasks by priority (P1 first). Each task shows estimated time, category color, checkbox. Overdue tasks shown in red with a delay badge.
	•	GoalProgressRail: Horizontal progress bars for all active goals. Color-coded: green ≥ 80%, amber 50–79%, red < 50%. Clicking a bar navigates to that goal's detail view.
	•	HabitStreak Widget: Grid of today's habits. Tap to mark complete. Shows current streak count below each icon.
	•	FocusTimer: Persistent timer widget. One-click to start a deep work session on the selected task. Persists across navigation.
	•	WeeklyScoreCard: Current week's productivity score with delta from last week (↑/↓). Breakdown tooltip on hover.
	•	DeadlineRadar: Next 7 days. Shows goals and tasks with approaching deadlines, color-coded by urgency.

Layout Rule
Responsive 3-column grid on desktop (tasks | goals + habits | score + deadlines). Single column on mobile. Dark mode default.

Module 2 — Goal Hierarchy Engine  /goals
The structural core of the entire system. Goals cascade across 7 levels. Every goal must be linked to a parent (except Vision).

Goal Cascade Structure
  VISION  (root, no parent)
     └── YEARLY GOAL
            └── HALF-YEARLY GOAL
                   └── QUARTERLY GOAL
                          └── MONTHLY MILESTONE
                                 └── WEEKLY TARGET
                                        └── DAILY TASKS  ← linked to tasks table

Behavior Requirements
	•	When a user creates a goal at any level, the system must prompt: 'Would you like to break this into [next level] targets?' If yes, open the AI breakdown modal.
	•	Progress at the daily task level auto-aggregates upward. Completing 3 of 5 daily tasks = 60% on parent weekly goal. Weekly goals feed into monthly, and so on recursively.
	•	Velocity status (on_track / at_risk / behind) is computed nightly via a cron job: compare current progress% against expected% given days elapsed vs. total days.
	•	The UI must expose three views: Tree View (hierarchical expand/collapse), Kanban View (columns per status), Timeline View (Gantt-style with target dates).
	•	Each goal card must display: title, category badge, progress bar, velocity indicator, target date, days remaining, and child goal count.

Module 3 — Smart Task Breakdown Engine  /goals/:id/breakdown
When a user inputs any goal or task, the AI engine suggests a structured decomposition. This is the primary AI touchpoint.

AI Prompt Contract
Send the following context to the AI API for each breakdown request:
  System: You are a productivity coach. Given a goal, return a JSON array
  of tasks with: title, category, estimatedMins, priority (P1-P4), phase
  (study/practice/review/assess). Group into weeks if goal > 7 days.
 
  User: Goal: '{goal_title}', Target date: '{date}', Available hours/day: {n}
  Existing tasks for this goal: [{...}]
  Return ONLY valid JSON. No markdown, no explanation.

Module 4 — Time Blocking Scheduler  /schedule
A day-view calendar with drag-and-drop task slots. The source of truth for what the user commits to doing and when.

Functional Requirements
	•	Render a full-day timeline (5 AM – 11 PM) in 30-minute increments. Pre-populate with the user's recurring block templates.
	•	Task cards from the today list are draggable into any time slot. Snaps to 30-min boundaries. Minimum block size: 30 min.
	•	Each time block stores: type, linked task_id (optional), start/end time, estimated_mins, and actual_mins (filled post-session).
	•	At end of day, show a recap: planned hours vs. actual hours by block type. Render as stacked horizontal bar chart.
	•	Week view must show block distribution per day as a compact heatmap — darker = more scheduled hours.

Module 5 — Habit & Routine Tracker  /habits
Track recurring habits with streak logic, completion heatmaps, and milestone alerts. Streaks are the primary motivation mechanic.

Streak Rules
	•	A habit is 'completed' for a day if marked done before midnight in the user's local timezone.
	•	Missing one day resets the streak to 0. There is no grace period — this is intentional (builds discipline).
	•	current_streak and longest_streak are maintained in the habits table and updated synchronously when a log is inserted.
	•	Milestone alerts at 7, 14, 21, 30, 60, 100, 365 consecutive days. Trigger a badge award + XP bonus.

Visualizations Required
	•	GitHub-style contribution heatmap: 52-week × 7-day grid. Cell color intensity = number of habits completed that day. Tooltip shows date + completion count.
	•	Weekly bar chart: For the last 8 weeks, show % of habits completed per week as stacked bars by habit.
	•	Streak leaderboard: All habits sorted by current streak, descending. Shows name, streak, and record streak.

Auto-generated report. No manual input required — system compiles and presents. User adds annotations.

	•	Goal progress delta: where each active goal stood Monday vs. Sunday
	•	Habit success rate per habit (completions / scheduled days)
	•	Productivity score breakdown (see Section 4)
	•	AI-generated insight: one paragraph analyzing the week's patterns and suggesting one adjustment
	•	Preview: 3 most important tasks auto-suggested for next Monday

	•	Goal completion % per category
	•	Time allocation analysis: actual hours spent per block type (pie chart + table)
	•	Trend analysis: is each goal's velocity improving or degrading?
	•	AI generates a 'Month in Review' summary + top 3 recommendations for next month


SECTION 4
Scoring & Gamification


4.1  Weekly Productivity Score
Computed every Sunday at midnight. Stored in review_sessions. Displayed on dashboard as a number (0–100) with grade label.

  Weekly Score = (Task Completion × 0.40)
               + (Goal Progress   × 0.30)
               + (Habit Consistency× 0.20)
               + (Deep Work Hours  × 0.10)
 
  Task Completion  = completed_tasks / scheduled_tasks × 100
  Goal Progress    = avg(progress_delta) across active goals
  Habit Consistency= avg(completion_rate) across active habits
  Deep Work Hours  = min(actual_deep_work_hrs / target_hrs, 1) × 100
 
  Grades:  90–100 = S (Elite) | 75–89 = A | 60–74 = B | 45–59 = C | <45 = D

SECTION 5
AI Productivity Coach


The AI coach is a persistent sidebar assistant. It has read access to the user's full goal tree, task history, habit logs, and review sessions. It must provide contextual, actionable coaching — not generic advice.

5.1  Trigger Conditions (Proactive)
Trigger
AI Action
A weekly goal has < 20% progress with 3 days remaining
Surface alert: 'Your Weekly Target X is at risk. Here is a 3-day catch-up plan.'
User opens app with no tasks scheduled for today
Auto-suggest top 3 tasks from pending backlog based on priority and deadlines.
Weekly score drops > 15 points from previous week
Trigger a check-in prompt: 'Your score dropped this week. Want to identify what blocked you?'
Habit streak broken for a tracked habit
Gentle re-commitment prompt the next morning with streak recovery messaging.
Monthly goal < 40% complete at mid-month
Rebalance suggestion: 'You need 3.2 hours/day for the rest of the month. Here is a revised schedule.'

5.2  On-Demand Capabilities
	•	'Break down this goal for me' → returns structured task tree with durations and phases
	•	'Schedule my week' → given hours available per day, distributes pending tasks into time blocks
	•	'What should I focus on today?' → ranks pending tasks by urgency × impact using priority + deadline proximity
	•	'How am I doing?' → generates a natural-language summary of the user's current progress across all active goals
	•	'I'm behind on X' → generates a specific catch-up plan with daily targets to close the gap before the deadline

SECTION 6
Focus Mode & Analytics


6.1  Focus Mode  /focus
Full-screen, distraction-free deep work environment. Single task displayed. All navigation hidden.
	•	Mode: Pomodoro (25+5) or Custom (user sets work/break duration)
	•	Task is pre-selected from the scheduler or chosen at session start
	•	Session logged to focus_sessions on completion — not on exit
	•	On session end: prompt for actual_mins, self-assessed focus quality (1–10), brief note
	•	Ambient sound options: white noise, rain, binaural beats (use royalty-free CDN sources)
	•	During session: keyboard shortcut (Esc) to pause; double-Esc to abandon

SECTION 7
UX Rules & Build Order


7.1  Non-Negotiable UX Rules
	•	Zero perceived loading: Use optimistic UI everywhere. Update local state immediately on user action. Sync to server in background. Roll back on error.
	•	Dark mode by default. Light mode toggle saved to users.preferences. Apply via CSS variables — never hard-coded colors.
	•	Mobile-first layout. All views must be fully functional on a 375px viewport. Dashboard collapses to stacked single column.
	•	Keyboard-first power user mode: Cmd+K opens command palette. Quick-add task via Cmd+T. Start focus mode via Cmd+F. Navigate modules via number keys 1–6.
	•	All destructive actions (delete goal, clear habit log) must require two-step confirmation. No undo after confirmation.
	•	Every data-heavy page must have an empty state that guides the user. No blank white screens.

The Core Principle
This system is not a to-do app. It is a trajectory management engine. The goal is not to schedule perfectly — schedules break. The goal is to create feedback loops so tight that deviation from your goals becomes immediately visible and immediately correctable. Build that, and the outcomes take care of themselves.
